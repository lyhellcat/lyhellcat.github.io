Configuration & set up

Git stores config options in 3 separete files.

  • Local: /.git/config repo特定的设置

  • Global: /.gitconfig 使用--global标志时修改的文件, 用户特定的设置

  • System: $(prefix)/etc/gitconfig 系统范围的设定

当这些文件中的选项发生冲突时, 本地设置会覆盖用户设置, 而用户设置会覆盖系统范围.

Saving change

Git是一个分布式应用架构, 提交可以在本地捕获和建立, 然后使用git push -u origin master.

git add

git add将工作目录中的更改添加到暂存区域. 但实际上git add并不会真正影响到repo, 只有执行git commit才会发生变化.

git commit

在Git中, 存储库时分布式的, 将快照提交到本地仓库并不需要和其他Git仓库交互, 之后这个快照(Snapshots)可被push到任意远程仓库.

git diff

默认显示自上次提交以来所有未提交的更改.

git stash

暂时存储对工作副本所做更改, 当执行push时, stashed changes并不会被推送到服务器.

git stash可用于以下情景:

  • 发现有一个类是多余的，想删掉它又担心以后需要查看它的代码，想保存它但又不想增加一个脏的提交。这时就可以考虑git stash。
  • 使用git的时候，我们往往使用分支（branch）解决任务切换问题，例如，我们往往会建一个自己的分支去修改和调试代码, 如果别人或者自己发现原有的分支上有个不得不修改的bug，我们往往会把完成一半的代码commit提交到本地仓库，然后切换分支去修改bug，改好之后再切换回来。这样的话往往log上会有大量不必要的记录。其实如果我们不想提交完成一半或者不完善的代码，但是却不得不去修改一个紧急Bug，那么使用git stash就可以将你当前未提交到本地（和服务器）的代码推入到Git的栈中，这时候你的工作区间和上一次提交的内容是完全一样的，所以你可以放心的修Bug，等到修完Bug，提交到服务器上后，再使用git stash apply将以前一半的工作应用回来。
  • 经常有这样的事情发生，当你正在进行项目中某一部分的工作，里面的东西处于一个比较杂乱的状态，而你想转到其他分支上进行一些工作。问题是，你不想提交进行了一半的工作，否则以后你无法回到这个工作点。解决这个问题的办法就是git stash命令。储藏(stash)可以获取你工作目录的中间状态——也就是你修改过的被追踪的文件和暂存的变更——并将它保存到一个未完结变更的堆栈中，随时可以重新应用。

.gitignore

Git将工作副本中的文件区分为 1) tracked, 之前add或commit的文件 2) untracked, 尚未暂存或提交的文件 3) ignored, 忽略

  • dependency caches, such as the contents of /node_modules or /packages

  • compiled code, such as .o, .pyc, and .class files

  • build output directories, such as /bin, /out, or /target

  • files generated at runtime, such as .log, .lock, or .tmp

  • hidden system files, such as .DS_Store or Thumbs.db

  • personal IDE config files, such as `.idea/workspace.xml

Reactor VS. Proactor

服务器程序通常需要处理3类事件: IO、信号及定时事件

  • Reactor模式: 主线程 (IO处理单元) 负责监听文件描述符上是否有事件发生 (可读, 可写), 如果有则立即通知工作线程将可读可写事件放入请求队列, 交给工作线程处理, 通常由同步IO实现
  • Proactor模式: 将所有IO都交给主线程和内核处理 (读, 写), 工作线程仅负责处理逻辑. e.g. 主线程读users[sockfd].read(), 选择工作线程处理客户请求pool->append(users + sockfd). 通常由异步IO实现

实际应用中常采用同步IO模拟Proactor模式

  • 主线程往epoll内核事件表注册socket上的读就绪事件

  • 主线程调用epoll_wait等待socket上有数据可读

  • 当socket上有数据可读，epoll_wait通知主线程,主线程从socket循环读取数据，直到没有更多数据可读，然后将读取到的数据封装成一个请求对象并插入请求队列

  • 睡眠在请求队列上某个工作线程被唤醒，它获得请求对象并处理客户请求，然后往epoll内核事件表中注册该socket上的写就绪事件

  • 主线程调用epoll_wait等待socket可写

  • 当socket上有数据可写，epoll_wait通知主线程。主线程往socket上写入服务器处理客户请求的结果

  • 半同步/半反应堆工作流程（以Proactor模式为例）

    • 主线程充当异步线程，负责监听所有socket上的事件
    • 若有新请求到来，主线程接收之以得到新的连接socket，然后往epoll内核事件表中注册该socket上的读写事件
    • 如果连接socket上有读写事件发生，主线程从socket上接收数据，并将数据封装成请求对象插入到请求队列中
    • 所有工作线程睡眠在请求队列上，当有任务到来时，通过竞争（如互斥锁）获得任务的接管权

epoll select poll 的选择

对于select和poll, 所有文件描述符都是在用户态被加入到文件描述符集合的, 调用时需要将整个集合拷贝到内核态. epoll直接将文件描述符维护在内核态, 添加文件描述符时都需要执行一个系统调用, 而系统调用的开销较大, 在很多短期活跃连接的情况下, epoll可能会慢于select和poll.

select使用线性表描述文件描述符集合, poll使用链表, epoll使用红黑树, 同时维护有ready list, 将事件表中已经就绪的事件添加到ready list中. 使用epoll_wait调用时, 只需要观察list中是否有数据即可.

select和poll的开销来自内核判断是否有文件描述符就绪. 执行select或poll调用时, 会采取遍历的方式, 判断是否有文件描述符活动. epoll当有活动产生时, 会自动触发epoll回调函数通知epoll文件描述符, 然后将就绪的文件描述符放到ready list中等待epoll_wait调用后处理.

当监测的fd数量较小, 且各个fd都很活跃时, 选择select和poll. 当fd较多, 且单位时间内仅部分fd活跃时, 使用epoll会明显提升性能.

epoll中LT(电平触发)和ET(边缘触发)的区别

  • LT（电平触发）：类似select，LT会去遍历在epoll事件表中每个文件描述符，来观察是否有我们感兴趣的事件发生，如果有（触发了该文件描述符上的回调函数），epoll_wait就会以非阻塞的方式返回。若该epoll事件没有被处理完（没有返回EWOULDBLOCK），该事件还会被后续的epoll_wait再次触发.
  • ET（边缘触发）：ET在发现有我们感兴趣的事件发生后，立即返回，并且sleep这一事件的epoll_wait，不管该事件有没有结束.

在使用ET模式时，必须要保证该文件描述符是非阻塞的（确保在没有数据可读时，该文件描述符不会一直阻塞）；并且每次调用read和write的时候都必须等到它们返回EWOULDBLOCK（确保所有数据都已读完或写完）

C++如何解析函数调用

Name lookup

Member name lookup, 当函数名位于.或是->右侧时使用Member name lookup. 例如foo->bar(), 该方式用于查找特定类成员.

Qualified name lookup, 当包含::时, 例如std::sort等显式指定namespace的, 将会在指定命名空间内寻找调用的函数.

Unqualified name lookup, 当编译器得到非限定名称时, 会根据上下文在不同的范围内寻找匹配的声明.

对于上例的blast(ast, 100);, 有3个候选函数, 第1个候选函数显示了一个特性argument-dependent lookup, ADL, 即使调用方并未指定namespace为galaxy, 编译器仍会在参数的命名空间中寻找候选的函数.

Special Handling of Function Templates

blast(ast, 1000)并未指定模板参数, 因此编译器需要使用模板参数推导的方式确定③中的类型T.

之后编译器将会进行模板参数替换.

在模板参数替换失败时, template function就会从候选列表中被删除. 在模板元编程中, 称为SFINAE, substitution faliure is not an error.

Overload Resolution 重载决议

在这一阶段函数模板已经消失, 只剩下一组整洁的待候选函数集合.

接下来的工作室进一步缩小这一集合, 确定那些函数可以处理该函数调用.

最明显的要求就是函数的参数列表需要兼容.

对于candidate②, 由于Target有一个转换构造函数Target(galaxy::Asteroid *ast) : ast{ast} {}, 因此第一个参数可隐式转换为Target类型, 但是参数个数不匹配. 可以使用explicit的方式进行显式标记避免隐式转换.

candidate③的参数列表与①一致, 因此也兼容.

Tiebreakers 决胜局

在上例中, 只剩下2个可供调用的函数, 但是他们的参数列表都符合条件, 能够处理原始的function call. 但是编译器必须确定哪个是最好的可行函数.

(1) Better-matching parameters wins / 参数匹配更高者获胜

C++最重视调用者的参数类型与函数参数类型的匹配程度, 总是选择从给定的参数进行较少隐式类型转换的函数. 例如对于std::vector的operator[], 将会决定调用const版本还是non const版本. 在示例中, 2个备选函数具有相同参数列表, 因此决策进入下一局.

(2) Non-template function wins / 非模板函数取胜

C++更倾向调用非模板函数而不是模板函数, 因此最终将会调用

(3) More specialized template wins / 更偏特化的模板函数取胜
