Configuration & set up

Git stores config options in 3 separete files.

  • Local: /.git/config repo特定的设置

  • Global: /.gitconfig 使用--global标志时修改的文件, 用户特定的设置

  • System: $(prefix)/etc/gitconfig 系统范围的设定

当这些文件中的选项发生冲突时, 本地设置会覆盖用户设置, 而用户设置会覆盖系统范围.

Saving change

Git是一个分布式应用架构, 提交可以在本地捕获和建立, 然后使用git push -u origin master.

git add

git add将工作目录中的更改添加到暂存区域. 但实际上git add并不会真正影响到repo, 只有执行git commit才会发生变化.

git commit

在Git中, 存储库时分布式的, 将快照提交到本地仓库并不需要和其他Git仓库交互, 之后这个快照(Snapshots)可被push到任意远程仓库.

git diff

默认显示自上次提交以来所有未提交的更改.

git stash

暂时存储对工作副本所做更改, 当执行push时, stashed changes并不会被推送到服务器.

git stash可用于以下情景:

  • 发现有一个类是多余的，想删掉它又担心以后需要查看它的代码，想保存它但又不想增加一个脏的提交。这时就可以考虑git stash。
  • 使用git的时候，我们往往使用分支（branch）解决任务切换问题，例如，我们往往会建一个自己的分支去修改和调试代码, 如果别人或者自己发现原有的分支上有个不得不修改的bug，我们往往会把完成一半的代码commit提交到本地仓库，然后切换分支去修改bug，改好之后再切换回来。这样的话往往log上会有大量不必要的记录。其实如果我们不想提交完成一半或者不完善的代码，但是却不得不去修改一个紧急Bug，那么使用git stash就可以将你当前未提交到本地（和服务器）的代码推入到Git的栈中，这时候你的工作区间和上一次提交的内容是完全一样的，所以你可以放心的修Bug，等到修完Bug，提交到服务器上后，再使用git stash apply将以前一半的工作应用回来。
  • 经常有这样的事情发生，当你正在进行项目中某一部分的工作，里面的东西处于一个比较杂乱的状态，而你想转到其他分支上进行一些工作。问题是，你不想提交进行了一半的工作，否则以后你无法回到这个工作点。解决这个问题的办法就是git stash命令。储藏(stash)可以获取你工作目录的中间状态——也就是你修改过的被追踪的文件和暂存的变更——并将它保存到一个未完结变更的堆栈中，随时可以重新应用。

.gitignore

Git将工作副本中的文件区分为 1) tracked, 之前add或commit的文件 2) untracked, 尚未暂存或提交的文件 3) ignored, 忽略

  • dependency caches, such as the contents of /node_modules or /packages

  • compiled code, such as .o, .pyc, and .class files

  • build output directories, such as /bin, /out, or /target

  • files generated at runtime, such as .log, .lock, or .tmp

  • hidden system files, such as .DS_Store or Thumbs.db

  • personal IDE config files, such as `.idea/workspace.xml

Undoing changes

在Git上执行undoing操作时, 通常是回到过去或是另一条没有发生错误的时间线, 因此可将Git视作时间轴管理工具.

使用git log查看提交信息后, 可使用git checkout来访问先前的提交

git revert是撤销shared public changes的最佳工具

git reset 适用于撤销本地的私有修改

git checkout 可移动并查看提交历史

常见面试题

git rebase和git merge的区别

在feature分支进行新特性的开发, 与此同时master分支上也有提交

执行merging

merge特点: 创建一个新的commit, 合并时候遇到冲突, 需要修改后重新commit

rebase的本质是变基, 需要查找二者的公共祖先

rebase特点: 会合并之前的commit历史, 去掉了merge commit.

缺点在于出现代码问题难以定位, 因为re-write了history
